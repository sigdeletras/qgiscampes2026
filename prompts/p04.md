## 1. Estrategia de prompts para calidad

### ✅ Prompt de alta calidad (ejemplo real para tu plugin)
```
Quiero una función en Python para QGIS que:
1. Descargue un ATOM feed de Catastro

3. Maneje errores de red y XML malformado

6. Siga PEP 8 (líneas <80 caracteres)
7. No exceda 50 líneas (si es más, sugiere dividirla)

No me des código con:

```


**Efecto**: Recibirás código mínimo, sin manejo de errores, difícil de mantener.

## 2. El proceso de "3 pasadas" (obligatorio)

| Paso | Qué haces | Herramienta IA | Resultado |
|------|-----------|----------------|------------|
| **1. Generar** | Pides código con especificaciones claras | Claude/GPT-4 | Borrador funcional |
| **2. Refactorizar** | "Refactoriza este código: divide funciones >30 líneas, añade typing hints, usa nombres más descriptivos" | Cualquiera | Código estructurado |
| **3. Documentar** | "Genera docstrings estilo Google/NumPy y un ejemplo de uso para cada función pública" | DeepSeek/Gemini | Código profesional |

## 4. Control de calidad: checklist para cada función

Antes de aceptar código generado, verifica:

### ✅ Señales de calidad
- [ ] **Type hints**: `def funcion(param: str) -> list:`
- [ ] **Docstring** explicando qué, por qué y ejemplo
- [ ] **Nombres reveladores**: `obtener_datos_municipio()` vs `proc()`
- [ ] **Manejo de errores específico**: `except ConnectionError:` no `except:`
- [ ] **Constantes al inicio**: `TIMEOUT = 30` no `requests.get(url, timeout=30)`
- [ ] **Separación de responsabilidades**: Una función hace UNA cosa

### ❌ Señales de alerta (pide refactor)
- [ ] Código >50 líneas sin dividir
- [ ] Variables: `data1`, `temp`, `x`
- [ ] Comentarios tipo `# incrementar contador` (el código ya lo dice)
- [ ] "Código muerto" comentado
- [ ] Sin `if __name__ == "__main__"` para pruebas

# Gemini

Para un principiante, el mayor riesgo de programar con IA no es que el código no funcione, sino el **"código Frankenstein"**: fragmentos de código que la IA genera de forma aislada y que acabas pegando sin entender del todo, creando un programa difícil de mantener, lleno de redundancias o propenso a fallar.

Para asegurar que tu complemento de QGIS tenga una calidad de la que te sientas orgulloso frente a otros desarrolladores, debes guiar a la IA utilizando buenas prácticas de ingeniería de software. Aquí tienes las reglas de oro para lograrlo:

---

## 1. Adopta una Arquitectura Limpia (Separa los conceptos)

No dejes que la IA meta todo el código en un solo archivo. Si le pides *"haz el plugin de Catastro"*, meterá las peticiones web, el parseo del XML, el diseño de la interfaz y la lógica de QGIS en el mismo sitio.

Obliga a la IA a separar el código en tres capas independientes:

* **Capa de Interfaz (`gui`):** Solo controla lo que pasa en la pantalla (ej. *"si el usuario hace clic aquí, activa este botón"*).
* **Capa de Servicio (`services/catastro_client.py`):** Una clase de Python puro que solo sabe hablar con el ATOM de Catastro, descargar el XML y devolver datos limpios. No sabe nada de QGIS.
* **Capa de QGIS (`core`):** Código exclusivo de PyQGIS para coger el archivo descargado y pintarlo en el mapa.

Al estar separado, si Catastro cambia su formato XML mañana, solo tendrás que tocar el archivo del servicio, y el resto del plugin seguirá funcionando intacto.

---

## 2. Establece las "Reglas de Estilo" en el Prompt

La IA se adapta al nivel de exigencia que le pidas. Antes de que empiece a escribir una sola línea de código, dale un contexto estricto de calidad.

> **Prompt de ejemplo:**
> *"Actúa como un desarrollador Senior de Python y QGIS. Vamos a escribir la función para parsear el XML de Catastro. Requisitos de calidad obligatorios:*
> * *Sigue estrictamente la guía de estilo **PEP 8**.*
> * *Usa **Type Hinting** (tipado de datos) en los argumentos y retornos de las funciones.*
> * *Incluye **Docstrings** claros en formato Google para explicar qué hace la función.*
> * *Evita funciones de más de 20 líneas; si es más larga, divídela en funciones auxiliares."*
> 
> 

### Comparativa de código generado por IA:

* **Mal código (Sin directrices):**
```python
def proc_xml(x):
    import xml.etree.ElementTree as ET
    root = ET.fromstring(x)
    # ... 40 líneas de código mezclado ...
    return urls

```


* **Buen código (Con directrices):**
```python
import xml.etree.ElementTree as ET
from typing import List

def extract_municipality_urls(xml_content: str) -> List[str]:
    """Parses the Catastro ATOM XML to extract download URLs.

    Args:
        xml_content: The raw XML string from the ATOM feed.

    Returns:
        A list of string URLs pointing to the zip files.
    """
    root = ET.fromstring(xml_content)
    return [entry.find('link').attrib['href'] for entry in root.findall('entry')]

```

## 3. Pide explicaciones ("Code Review") antes de integrar

Nunca copies y pegues código que no entiendas. Usa a la IA como un tutor que revisa su propio trabajo. Cuando te devuelva una función compleja (como el parseo del árbol XML), hazle estas preguntas:

* *"¿Por qué has elegido esta estructura o librería en lugar de esta otra?"*
* *"Explícame línea por línea qué hace este bucle."*
* *"¿Qué posibles fallos (excepciones) puede tener este código si el servidor de Catastro se cae, y cómo los hemos controlado?"*

Si un desarrollador ve que tu código maneja bien los errores (`try/except`) cuando no hay conexión a internet, sabrá que es un código maduro, aunque lo haya escrito un principiante apoyado en IA.

---

## 4. Automatiza la revisión (Linters y Formateadores)

No confíes al 100% en el formato que te escupa el chat de la IA. Configura tu editor de código (VS Code o Cursor) con dos herramientas estándar de Python que corrigen el estilo visual de forma automática al guardar el archivo:

* **Black:** Un formateador que ordena los espacios, saltos de línea y comillas para que el código parezca escrito por la misma persona.
* **Flake8 / Ruff:** Herramientas que analizan el código en busca de variables que no usas, librerías importadas de más o errores de sintaxis comunes.

Si pasas tu código por estas herramientas antes de enseñárselo a alguien, visualmente será idéntico al de un desarrollador profesional.

# Claude

Pregunta clave, y más importante de lo que parece al principio. La IA genera código que *funciona* mucho antes de generar código que *se entiende*. Son cosas distintas.

---

## El problema de fondo

Cuando la IA genera código y tú no lo entiendes del todo, tienes un problema silencioso: el plugin funciona hoy, pero en tres meses ni tú ni nadie sabrá por qué hace lo que hace. Y si falla, no sabrás por dónde empezar.

La regla más importante es: **no aceptes código que no puedas explicar en voz alta**. Si no puedes decir "esta función hace X porque Y", pídele a la IA que te lo explique antes de seguir.

---

## Prácticas concretas, por área

### Estructura y organización

Separa responsabilidades desde el principio. Un fichero no debería hacer todo:

```
mi_plugin/
├── __init__.py
├── plugin.py          # Solo punto de entrada QGIS
├── dialog.py          # Solo UI
├── downloader.py      # Solo lógica de descarga HTTP
├── atom_parser.py     # Solo parseo del feed ATOM
└── layer_loader.py    # Solo carga de capas en QGIS
```

Cuando le pidas código a la IA, hazlo módulo a módulo, no "dame el plugin entero". Así entiendes cada pieza antes de ensamblar.

### Nomenclatura

Pídele explícitamente a la IA nombres descriptivos. Hay diferencia entre:

```python
# Lo que la IA puede generar si no le dices nada
def proc(x, lst):
    for i in lst:
        if i[0] == x:
            return i[1]

# Lo que debes exigir
def find_municipality_url(municipality_name: str, feed_entries: list) -> str:
    for entry in feed_entries:
        if entry["name"] == municipality_name:
            return entry["download_url"]
```

El segundo se lee como prosa. Si ves variables de una letra o nombres como `data2`, pide refactoring antes de aceptar.

### Docstrings y comentarios

Añade al prompt: *"incluye docstring en cada función explicando qué recibe, qué devuelve y posibles excepciones"*. Ejemplo de lo que debes conseguir:

```python
def download_and_extract(url: str, temp_dir: str) -> str:
    """
    Descarga un ZIP desde url, lo extrae en temp_dir
    y devuelve la ruta del fichero GML encontrado.

    Args:
        url: URL directa al fichero ZIP de Catastro
        temp_dir: Directorio temporal donde extraer

    Returns:
        Ruta absoluta al fichero .gml extraído

    Raises:
        ValueError: Si no se encuentra ningún .gml en el ZIP
        requests.HTTPError: Si la descarga falla
    """
```

Esto no es burocracia: es lo que hace que otro desarrollador (o tú en seis meses) no tenga que leer el código para entender qué hace una función.

### Manejo de errores

La IA tiende a generar el "camino feliz" y omitir errores. Para tu plugin, el feed puede no responder, el ZIP puede estar corrupto, el municipio puede no tener datos. Pide siempre:

```python
# En lugar de esto (lo que la IA genera por defecto)
response = requests.get(url)
data = response.content

# Exige esto
try:
    response = requests.get(url, timeout=30)
    response.raise_for_status()
except requests.Timeout:
    raise RuntimeError(f"Tiempo de espera agotado descargando {url}")
except requests.HTTPError as e:
    raise RuntimeError(f"Error HTTP {e.response.status_code} en {url}")
```

## Herramientas que hacen el trabajo de revisión automático

Instala estas tres en VS Code o Cursor, una vez, y te avisarán de problemas sin que tengas que pensar en ello:

**Ruff** — linter y formateador ultrarrápido. Detecta código no usado, imports mal ordenados, estilo inconsistente. Un fichero `ruff.toml` mínimo en tu proyecto es suficiente.

**Pylance** — análisis de tipos estático. Si la IA genera una función que espera `str` y le pasas `None`, Pylance lo subraya antes de ejecutar.

**Pre-commit** — ejecuta Ruff automáticamente antes de cada `git commit`. Así el código sucio nunca llega al repositorio.

No necesitas configurarlos a fondo. Con los valores por defecto ya capturan el 80% de los problemas habituales.

---

## El hábito más valioso: el prompt de revisión

Antes de dar por bueno cualquier bloque generado, pásalo por este prompt:

> *"Revisa este código como si fuera un code review para un desarrollador junior. Señala: funciones demasiado largas, nombres poco descriptivos, errores no manejados, lógica duplicada y cualquier cosa que un desarrollador nuevo no entendería sin contexto adicional."*

La IA encuentra sus propios problemas con sorprendente eficacia cuando se lo pides explícitamente. Es distinto a pedirle que genere: aquí actúa como revisor.

---

## Lo que no te dará ninguna herramienta

La legibilidad final depende de que **tú entiendas el dominio**: qué hace QGIS con una capa vectorial, cómo funciona el feed ATOM de Catastro, qué significa cada parámetro. Una función puede estar perfectamente documentada y aun así ser una mala solución al problema si quien la generó (la IA) no entendió bien el contexto.

Por eso el mejor control de calidad eres tú leyendo el código generado con la documentación de PyQGIS abierta al lado, no delegando esa lectura en otra herramienta.
