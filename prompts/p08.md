# Chatgpt

Para este tipo de plugin QGIS (descarga INSPIRE Catastro + ATOM + carga en QGIS), los factores críticos no suelen estar en la IA ni en Python básico, sino en **integración SIG, robustez de datos y mantenibilidad del flujo geoespacial**.

A nivel práctico, hay varios puntos que conviene cerrar antes de seguir desarrollando.

---

# 1. Modelo de datos real (lo más importante)

Antes de código, necesitas fijar esto:

* Qué geometría vas a tratar en cada capa:

  * Parcelas → polígonos
  * Edificios → polígonos
  * Direcciones → puntos o líneas (depende del GML)
* CRS esperado (INSPIRE suele venir en ETRS89 / EPSG:4258 o variantes)
* Volumen de datos por municipio (impacta rendimiento)
* Si habrá filtrado espacial o solo carga directa

Si esto no está claro, el plugin se vuelve inestable rápido.

---

# 2. Estrategia de carga en QGIS (decisión crítica)

Tienes 3 enfoques posibles:

### A. Carga directa GML (rápido, simple)

* `QgsVectorLayer(path, "name", "ogr")`

✔ fácil
✖ lento con grandes municipios
✖ problemas de CRS/validación

---

### B. Conversión a GeoPackage (recomendado a medio plazo)

Pipeline:

1. descargar GML
2. convertir a GPKG (GDAL)
3. cargar GPKG

✔ rendimiento estable
✔ mejor indexación
✔ reproducible
✖ más pasos

---

### C. Capa virtual / streaming

* más complejo
* útil solo si escalas mucho

---

# 3. Gestión de ATOM INSPIRE (punto crítico real)

El mayor riesgo del proyecto no es QGIS, es esto:

* feeds anidados
* links indirectos
* variabilidad por provincia
* XML inconsistente o cambiante

Debes asumir:

* el ATOM NO es una API estable tipo REST
* es un grafo de enlaces XML

Recomendación:

✔ encapsular todo en un `AtomResolver`
✔ nunca mezclar lógica ATOM con UI

---

# 4. Estrategia de errores (muy importante en GIS)

Debes diseñar desde el inicio:

* logging claro (QgsMessageLog)
* fallback si un municipio falla
* retry de descargas
* validación de XML corrupto
* manejo de capas vacías

En GIS real, el 30–40% de casos son “edge cases”.

---

# 5. Rendimiento y UX en QGIS

QGIS se bloquea fácilmente si no controlas:

* descargas síncronas
* parsing pesado en hilo principal
* carga de capas grandes

Necesitas:

* QThread o QgsTask para descargas
* barra de progreso real
* cancelación de tareas

---

# 6. Separación estricta UI / lógica SIG

Error típico en plugins:

* UI llama directamente a HTTP
* UI parsea XML
* UI carga capas

Esto hay que evitar.

Arquitectura mínima:

```text
UI (PyQt)
  ↓
Controller
  ↓
Services (ATOM / Download / QGIS Loader)
  ↓
QGIS API
```

---

# 7. Dependencia GDAL/OGR (subestimada)

El plugin depende indirectamente de:

* versión GDAL del usuario
* drivers GML
* configuración de QGIS

Debes asumir:

* diferentes comportamientos entre QGIS 3.x

---

# 8. Normalización INSPIRE (problema oculto)

INSPIRE no es homogéneo:

* nombres de campos pueden variar
* namespaces XML distintos
* estructuras GML complejas

Recomendación:

✔ crear un “mapeador interno” de atributos
✔ no exponer directamente esquema INSPIRE al usuario

---

# 9. Estrategia de escalabilidad del plugin

Define desde el principio:

* ¿solo Catastro INSPIRE o más fuentes INSPIRE?
* ¿solo España o extensible a otros países?
* ¿descarga puntual o sincronización?

Esto afecta toda la arquitectura.

---

# 10. Control de complejidad (muy importante con IA)

El mayor riesgo con IA es:

* sobrearquitectura (demasiadas capas)
* abstracciones innecesarias
* patrones “enterprise” prematuros

Regla práctica:

> si no mejora rendimiento, testabilidad o claridad → no se añade

---

# 11. Reproducibilidad GIS

Debes asegurar:

* mismos resultados en diferentes máquinas
* rutas temporales controladas
* limpieza de archivos descargados
* compatibilidad con proyectos QGIS

---

# 12. Estrategia de desarrollo recomendada

Orden óptimo:

1. Resolver ATOM → obtener URL GML
2. Descargar fichero
3. Cargar capa en QGIS
4. UI básica
5. Paralelización (QThread/QgsTask)
6. Cache local
7. Conversión a GeoPackage (si es necesario)

---

## 1. Primer acontecimiento inesperado: el servicio WFS con límite por área

Aunque el Catastro ofrece un servicio WFS estándar, tiene una **restricción crítica**: las peticiones GetFeature no pueden superar **1 km² de extensión** para parcelas .

**Por qué esto importa para tu plugin**:
- Si un usuario abre el plugin y pide "todas las parcelas de Madrid capital", el servidor devolverá un error "Area of extension out of limits" 
- El usuario pensará que tu plugin está roto
- **Tú necesitas gestionar este error** y guiar al usuario a seleccionar áreas más pequeñas

**Consejo práctico**: Dado que ya planeas usar descarga ATOM por municipio (no WFS directo), este problema **no te afectará directamente** porque los ATOM descargan el municipio completo. Pero si en el futuro amplías el plugin a descarga por área de interés, ten esto en cuenta.

---

## 2. Segundo acontecimiento inesperado: navegadores que no interpretan ATOM

Esto viene directamente de la documentación oficial del Catastro: "Existen algunos navegadores que no interpretan bien los ficheros ATOM y no permiten realizar la descarga, como por ejemplo **Chrome**" .

**Implicación para tu plugin**:
- Tu plugin usa peticiones HTTP directas, no un navegador. Así que en principio no tendrás este problema.
- Pero si en el futuro usas el navegador web de QGIS para algo relacionado, ten cuidado.

---

## 3. Tercer acontecimiento inesperado: la estructura de dos niveles del ATOM

El feed ATOM principal no contiene los links de descarga directa. Tiene una arquitectura de dos niveles :

```
Nivel 1 (feed principal):
https://www.catastro.hacienda.gob.es/INSPIRE/CadastralParcels/ES.SDGC.CP.atom.xml
  └── Entries por gerencia (códigos GG)

Nivel 2 (feed de gerencia):
http://www.catastro.minhap.es/INSPIRE/CadastralParcels/GG/ES.SDGC.AD.atom_GG.xml
  └── Aquí están los links de descarga reales por municipio
```

**Lo que debes implementar**:
1. Primero parsear el feed principal para obtener los códigos de gerencia (GG)
2. Para cada gerencia, acceder a su feed específico
3. En ese feed de segundo nivel, buscar el municipio concreto y obtener su URL de descarga

## 5. Datos descargables: parcelas urbanas vs rústicas

Según la documentación técnica, los datasets de parcelas se generan **separados por tipo de cartografía** (urbana y rústica) :

```
Nomenclatura del fichero: A.E.S.SDGC.CP.R.GGMMM
Donde R = U (Urbana) o R (Rústica)
```

**Implicación para tu plugin**:
- Para un mismo municipio, puede haber dos archivos ZIP separados
- Tu plugin debería permitir descargar uno, otro, o ambos
- La interfaz debe reflejar esta distinción

---

## 6. Formatos de los datos descargados

Los archivos ZIP contienen :
- Un fichero **XML** con metadatos del dataset (incluye "MD" en el nombre)
- Un fichero **GML** con la información geográfica real

El GML de parcelas contiene dos tipos de objetos :
- **CPCadastralParcel**: la parcela individual
- **CPCadastralZoning**: la agrupación (manzana/polígono)

**Para QGIS**: Al cargar el GML, se añadirán dos capas (parcelas y zonas). Tu plugin debe gestionar esto adecuadamente.

---

## 7. Otra herramienta oficial del Catastro: Check4SEC

La Dirección General del Catastro ha desarrollado **Check4SEC**, un plugin de QGIS oficial para generar GMLs válidos para la Sede Electrónica del Catastro .

**Por qué es relevante**:
- Es un plugin **oficial** (desarrollado por la propia Dirección General del Catastro)
- Utiliza los mismos servicios de descarga INSPIRE que tú usarás 
- Si tu plugin necesita subir datos validados al Catastro, Check4SEC es la referencia

Esto confirma que el Catastro **apoya y fomenta** el uso de QGIS con sus servicios INSPIRE.

# Gemini

Para que tu primer complemento de QGIS pase de ser un experimento que funciona en tu ordenador a un plugin robusto, estable y profesional, hay **cuatro aspectos críticos** que la IA no va a prever a menos que tú se los exijas.

Al venir del desarrollo web, muchos de estos conceptos te sonarán, pero su aplicación en el entorno de escritorio de QGIS tiene ciertas peculiaridades que debes tener en cuenta antes de picar la primera línea de código:

---

## 1. La regla de oro: No congeles la interfaz de QGIS (Hilos de ejecución)

Este es el error número uno de todos los principiantes. Cuando descargas un archivo pesado de internet (como un GML de un municipio entero) usando una petición web estándar (`requests.get`), Python detiene la ejecución de todo el programa hasta que la descarga termina. En una web usas promesas o *async/awats*, pero en QGIS, **si haces esto, toda la aplicación de QGIS se quedará "congelada" (en blanco y con el cursor cargando)**, lo que frustra mucho al usuario.

### Cómo solucionarlo con IA:

Debes pedirle a Claude que implemente la descarga utilizando **`QThread` y `QNetworkAccessManager**` (las herramientas nativas de Qt para segundo plano) o la clase **`QgsTask`** de QGIS.

* **El prompt clave:** *"Necesito que la descarga del archivo ZIP de Catastro se ejecute como una tarea en segundo plano usando `QgsTask`, mostrando una barra de progreso en la interfaz para que QGIS no se congele mientras descarga"*.

---

## 2. El sistema de coordenadas (CRSs) y Catastro

Los datos de Catastro de España se distribuyen en sistemas de coordenadas muy específicos. Generalmente, para la península se utiliza **EPSG:25830 (ETRS89 / UTM zone 30N)** (o las zonas 29 y 31 según la provincia) y **EPSG:4083** para Canarias.

Si el usuario de tu plugin está trabajando en un proyecto en un mapa de Google (EPSG:3857) o coordenadas geográficas mundiales (EPSG:4326), al cargar la capa de Catastro podría "caer" en el lugar equivocado si no gestionas la proyección.

### Cómo solucionarlo con IA:

Asegúrate de que el plugin verifique el Sistema de Referencia de Coordenadas (CRS) del proyecto activo.

* **El prompt clave:** *"Cuando cargues la capa GML descargada en QGIS usando `QgsVectorLayer`, asegúrate de definir explícitamente su CRS original (EPSG:25830) y comprueba si es necesario que QGIS la reproyecte al CRS del proyecto activo de forma automática"*.

---

## 3. Manejo estricto de errores (El servidor de Catastro falla)

Los servicios oficiales de la Sede Electrónica del Catastro son fantásticos, pero a veces sufren caídas por mantenimiento, cambian las URLs del ATOM sin previo aviso, o el municipio que el usuario busca no devuelve datos de "Edificios" porque es una zona rústica sin construcciones. Si tu código no anticipa esto, el plugin lanzará un error crítico de Python en la pantalla del usuario.

### Cómo solucionarlo con IA:

Exígele a Claude un sistema de control de excepciones robusto. Cada petición web y cada lectura de XML debe estar envuelta en bloques `try-except`.

* **El prompt clave:** *"Añade control de excepciones. Si el servidor de Catastro devuelve un código HTTP que no sea 200, o si el XML está vacío, captura el error y utiliza `QgsMessageLog` o un `QMessageBox` para mostrar un aviso amigable al usuario en lugar de romper el plugin"*.

---

## 4. La codificación y los nombres de archivos (El peligro de las tildes)

Estás trabajando con datos de municipios de España. Te vas a encontrar con nombres como *"San Cristóbal de la Cuesta"* o *"Castelló"*. Python 3 gestiona bien UTF-8, pero los sistemas operativos (especialmente Windows) y las librerías viejas de descompresión de ZIP a veces rompen los nombres de los archivos si contienen tildes, eñes o caracteres autonómicos.

### Cómo solucionarlo con IA:

Cuando manejes las rutas de los archivos temporales donde descargas los ZIP de Catastro, dile a la IA que limpie los nombres de los archivos antes de guardarlos en el disco local o que use el ID del municipio (código INE/Catastral) en lugar del nombre de texto.

* **El prompt clave:** *"Para guardar los archivos temporales en el disco, usa el código catastral/INE del municipio en lugar de su nombre de texto para evitar problemas con tildes, espacios o caracteres especiales en las rutas de Windows"*.

## Lo que conviene revisar antes de empezar

### 1. Conoce el territorio antes de codificar

Dedica 30 minutos a inspeccionar manualmente los feeds ATOM en el navegador antes de escribir nada. Abre estas URLs y observa la estructura XML:

- `https://www.catastro.hacienda.gob.es/INSPIRE/CadastralParcels/ES.SDGC.CP.atom.xml`

Verás que el feed raíz lista entradas por provincia, cada una con su propia URL de sub-feed. Descarga manualmente un ZIP de un municipio pequeño, extráelo y mira qué ficheros contiene. Cuántos GML hay, cómo se llaman, si hay ficheros auxiliares.

Este paso te ahorra horas de debugging después. La IA no conoce el comportamiento real de estos feeds mejor que tú inspeccionándolos directamente.

### 2. Elige una versión mínima de QGIS y no la cambies

Define ahora: *"este plugin requiere QGIS 3.28 LTR o superior"*. Esa decisión afecta qué clases y métodos puedes usar. QGIS 3.28 es la LTR actual más extendida en administraciones públicas españolas, que es tu público probable. Escríbelo en el `metadata.txt` del plugin desde el primer día.

### 3. El `metadata.txt` es más importante de lo que parece

QGIS lee este fichero para validar el plugin antes de cargarlo. Un error aquí y el plugin no arranca, sin mensaje claro. Revisa la [especificación oficial](https://docs.qgis.org/latest/en/docs/pyqgis_developer_cookbook/plugins/plugins.html#plugin-metadata) y rellénalo correctamente desde el principio:

```ini
[general]
name=Catastro INSPIRE Downloader
qgisMinimumVersion=3.28
description=Descarga datos INSPIRE de Catastro por municipio
version=0.1
author=Tu nombre
email=tu@email.com
```

### 4. Gestión de dependencias externas

Si usas `requests` en lugar de `urllib` (más cómodo pero no incluido en QGIS), necesitas gestionarlo. QGIS incluye su propio Python y no todas las librerías están disponibles. Las opciones son: usar solo la librería estándar (`urllib`, `zipfile`, `xml.etree`) o incluir un mecanismo de instalación de dependencias en el plugin. Para un primer plugin, **quédate con la librería estándar**. Es suficiente y evita un problema innecesario.

### 5. Dónde escribir ficheros temporales

No escribas en rutas absolutas ni en el directorio del plugin. QGIS tiene una forma correcta de obtener un directorio temporal:

```python
import tempfile
temp_dir = tempfile.mkdtemp()
```

O usando la API de QGIS si quieres integrarte mejor con el sistema de procesamiento. Decidirlo ahora evita refactoring después.

### 6. El Plugin Reloader es tan importante como el propio código

Instálalo antes de escribir la primera línea. Sin él, cada cambio en el código requiere reiniciar QGIS completo, lo que convierte el ciclo de desarrollo en algo insoportablemente lento. Con él, un atajo de teclado recarga el plugin en segundos.

## Una cosa que suele sorprender a los principiantes

Los errores de Python en plugins QGIS no siempre aparecen en un lugar obvio. Aprende desde el primer día a abrir la **consola de Python de QGIS** (`Plugins → Python Console`) y el **visor de registros** (`Ver → Paneles → Registro de mensajes`). Son tus herramientas de debugging principales, más útiles que el debugger de VS Code para la mayoría de errores del día a día.
