## 1. GPT-5.5

Muy fuerte en:

* Razonamiento.
* Diseño de arquitectura.
* Explicación de APIs.
* Depuración.
* Comprensión de proyectos GIS.

Para QGIS suele funcionar muy bien cuando le proporcionas:

* estructura del plugin,
* trazas de error,
* fragmentos de código PyQGIS.

Ejemplo:

```python
layer = QgsVectorLayer(path, name, "ogr")
```

Si falla, suele identificar mejor que otros modelos si el problema es:

* CRS,
* driver OGR,
* carga del GML,
* geometrías inválidas,
* rutas temporales.

# Deepseek

Basado en la experiencia de desarrolladores que han creado plugins para QGIS y en mi análisis, **alternar entre dos o tres LLMs según la tarea es la estrategia más efectiva**, especialmente para alguien con tu perfil (Python básico + experiencia web). No hay un modelo único que sea el mejor para todo el ciclo de desarrollo.

## Comparativa de LLMs para tu plugin INSPIRE

| LLM | Precio | Lo mejor para... | Puntos débiles |
|-----|--------|-----------------|----------------|
| **Claude 3.5 Sonnet** | Pago (~$20/mes vía API) | Arquitectura del plugin, código limpio, explicaciones didácticas, manejo de APIs externas (ATOM)  | Velocidad ligeramente menor en respuestas largas |
| **GPT-4 (o1 / Turbo)** | Pago (~$20/mes) | Depuración de errores complejos, parsing de XML/GML, tareas que requieren mucha precisión | Puede "sobrepensar" soluciones simples |
| **DeepSeek Chat v3.1** | **Gratis** / Muy bajo costo | Tareas repetitivas (estructura de carpetas, boilerplate), consultas simples de PyQGIS  | Menor calidad en código muy especializado |
| **Gemini 2.0 Flash** | Freemium (gratis con límites) | Prototipado rápido, preguntas conceptuales, búsqueda de documentación | Generación de código menos fiable que Claude/GPT-4  |

## Estrategia recomendada: combinación por tareas

Basándome en cómo funcionan los agentes especializados (como GeoAgent, que utiliza múltiples LLMs para diferentes subtareas ), te propongo este flujo:

### Fase 1: Diseño y arquitectura → **Claude 3.5 Sonnet**
- **Por qué**: Excepcional explicando conceptos y generando código limpio desde cero.
- **Prompt sugerido**: *"Quiero crear un plugin QGIS que descargue datos INSPIRE de Catastro. Dame la estructura de carpetas, el código del `plugin.py` principal y el archivo `metadata.txt`"*.
- **Resultado**: Obtendrás una base sólida y bien documentada.

### Fase 2: Implementación de funcionalidades específicas → **GPT-4 + DeepSeek**
- **Usa GPT-4 para**: Parsear el XML de los ATOM, manejar la API de descarga, cargar GML en QGIS con `addVectorLayer`.
- **Usa DeepSeek (gratis) para**: Código repetitivo (barra de progreso, manejo de errores básico, interfaz simple) .
- **Técnica**: Pídele a DeepSeek que genere el "esqueleto" y luego usa GPT-4 para refinar las partes críticas.

### Fase 3: Depuración y optimización → **Claude o GPT-4**
- **Por qué**: Ambos son superiores identificando errores contextuales .
- **Estrategia**: Copia el error completo y el fragmento de código. *"Este código falla con el error X. ¿Cómo lo arreglo manteniendo la compatibilidad con QGIS 3.x?"*

### Fase 4: Documentación y pulido → **Gemini 2.0 Flash (gratis)**
- **Por qué**: Rápido y suficiente para explicar funciones, generar comentarios, y sugerir mejoras de estilo.

## Herramientas que integran múltiples LLMs (recomendación TOP)

En lugar de pagar suscripciones separadas, usa una plataforma que te dé acceso a varios modelos:

| Herramienta | Modelos incluidos | Precio | Ideal para... |
|-------------|------------------|--------|----------------|
| **Cursor** | Claude, GPT-4, Gemini | $20/mes | Desarrollo diario, edición inline, contexto del proyecto completo  |
| **OpenRouter** | 200+ modelos (pago por uso) | ~$5-10/mes si usas poco | Alternar libremente entre modelos según la tarea  |
| **Promptly (plugin QGIS)** | Ollama, OpenAI, Anthropic | Gratis (necesitas tus claves API) | Ejecutar código LLM-generado directamente en QGIS  |

**Mi recomendación**: Paga **Cursor Pro** ($20/mes). Te da acceso a Claude y GPT-4 dentro de un editor que entiende todo tu proyecto . Además, su función `Cmd+K` para editar inline es perfecta para iterar rápido en el código del plugin .

## Ejemplo práctico de alternancia

**Tarea**: Parsear el ATOM de Catastro para extraer la URL de descarga por municipio.

1. **Pídele a Claude** (en Cursor): *"Dame una función Python que dado un nombre de municipio y un feed ATOM, devuelva la URL de descarga del GML. Usa `xml.etree`."*
   - Claude genera código limpio con manejo de namespaces.

2. **Si falla al ejecutarse**, copias el error a **GPT-4**: *"Esta función falla con `KeyError: '{}entry'`. ¿Cómo arreglo la búsqueda en el XML?"*
   - GPT-4 identifica el problema de namespaces y propone la corrección.

3. **Para la barra de progreso** (tarea repetitiva), usa **DeepSeek** (gratis vía OpenRouter): *"Genera una barra de progreso con QProgressDialog para una descarga larga en QGIS."*
   - DeepSeek es suficiente y no gastas en lo sencillo.

## ¿Y los LLMs gratuitos exclusivos?

**No recomiendo depender solo de gratuitos** para este proyecto, por tu nivel de experiencia. Los modelos gratuitos (Gemini Free, ChatGPT 3.5, Copilot Free) tienen tasas de acierto del ~70-75% frente al ~95% de los de pago . La diferencia se traduce en horas de depuración que pueden matar tu motivación.

**Excepción**: Usa **DeepSeek Chat v3.1** (gratuito y de calidad media-alta) para tareas muy acotadas y bien definidas . Pero no para el esqueleto principal del plugin.

## Resumen ejecutivo

| Tu situación | Estrategia de LLM |
|--------------|-------------------|
| Presupuesto $20/mes | **Cursor Pro** → acceso a Claude + GPT-4 en un solo editor |
| Presupuesto $0, pero con tiempo | **DeepSeek** (gratis) para estructura + **Gemini Free** para consultas. Asume 2-3x más tiempo de depuración. |
| Quieres lo mejor | **Cursor + OpenRouter** → usas Cursor para el día a día y OpenRouter para experimentar con modelos especializados (ej. para el parsing complejo) |

**En tu caso (Python básico, experiencia web)**: Empieza con la prueba gratuita de **Cursor** (14 días). Alterna entre Claude (para diseño y código nuevo) y GPT-4 (para depuración). Si te gusta, paga los $20/mes. Es la inversión más pequeña para garantizar que termines el plugin.

# Gemini

Para desarrollar este complemento de QGIS, **lo ideal es alternar modelos (o usar uno muy polivalente como "cerebro principal")**, ya que las tareas cambian mucho a lo largo del proyecto. No necesitas el mismo razonamiento para diseñar la arquitectura del plugin que para picar código repetitivo de la interfaz de usuario.

Actualmente, los LLMs se dividen en tres grandes "especialidades" para programación. Aquí tienes cuáles usar y cómo alternarlos según la fase de tu desarrollo:

---

## 🧠 1. El "Arquitecto y Solucionador" (Para lógica compleja de PyQGIS)

Esta es la fase donde necesitas que la IA entienda el flujo del ATOM de Catastro y cómo integrarlo en QGIS sin romper los hilos de ejecución de la interfaz (para que no se congele QGIS al descargar).

* **Modelos recomendados:** **Claude 3.5 Sonnet** (Anthropic) u **OpenAI o1 / o3-mini** (modelos de razonamiento).
* **¿Para qué usarlo?:**
* Diseñar la lógica de cómo leer el XML del ATOM de Catastro (que es un árbol de enlaces algo lioso).
* Escribir las funciones de descarga y descompresión del ZIP/GML.
* Resolver errores complejos de PyQGIS (cuando el *log* de QGIS te suelte un error de C++ que no entiendas).


* **Por qué destaca:** Son los mejores manteniendo la coherencia en scripts largos y aplicando patrones de diseño limpios.

## ⚡ 2. El "Copiloto Veloz" (Para autocompletado y código de la interfaz)

Cuando estás conectando los desplegables de tu interfaz de Qt Designer con las funciones de Python, no necesitas un modelo que "piense" mucho, sino uno que sea ultrarrápido y prediga lo que vas a escribir.

* **Modelos recomendados:** **Cursor Shampless (o sus modelos rápidos nativos)**, **Gemini 1.5 / 2.0 Flash**, o **GPT-4o mini**.
* **¿Para qué usarlo?:**
* Escribir el código repetitivo: rellenar los `QComboBox` con las provincias, limpiar los campos de texto, activar o desactivar botones.
* Autocompletar líneas de código mientras escribes en el editor (inline completion).


* **Por qué destaca:** Son modelos económicos, con ventanas de contexto enormes y una velocidad de respuesta casi instantánea.

---

## 🛠️ Cómo combinarlos en tu flujo de trabajo (Estrategia Práctica)

Si utilizas un entorno como **Cursor**, **Windsurf** o **VS Code**, puedes alternar de forma muy sencilla según lo que estés haciendo:

| Fase del Plugin | Tarea Específica | Modelo Ideal | Estrategia de Uso |
| --- | --- | --- | --- |
| **1. Inicio** | Estructurar el esqueleto del plugin y los archivos base. | **Claude 3.5 Sonnet** | Pídele que actúe como un desarrollador Senior de QGIS y te dibuje el mapa de archivos. |
| **2. Interfaz (UI)** | Mapear los elementos de Qt Designer (`.ui`) a variables de Python. | **GPT-4o mini** o **Gemini Flash** | Dale el archivo `.ui` y pídele que genere los conectores de los botones (`.clicked.connect`). Es rápido y barato. |
| **3. Lógica ATOM** | Parsear los XML de Catastro y descargar los GML por municipio. | **Claude 3.5 Sonnet** o **o3-mini** | Pídele que cree una clase aislada en Python para gestionar las peticiones HTTP y el XML de Catastro. |
| **4. Depuración** | QGIS se cierra o da un error crítico al cargar la capa en el mapa. | **Modelos de razonamiento (o1 / o3-mini)** | Pega el código de tu función de carga, el error del *log* de QGIS y deja que deduzca si es un problema de tipos de PyQGIS o de codificación del GML. |

**Mi recomendación para empezar hoy:** Configura tu editor con **Claude 3.5 Sonnet** como modelo por defecto para el chat (donde le haces preguntas y le pides funciones enteras) y un modelo más ligero (como **GPT-4o mini** o el nativo del editor) para el autocompletado en tiempo real mientras picas código.

## Los modelos más útiles para PyQGIS hoy

**Claude Sonnet 4.5 / Claude 3.7 Sonnet**
El punto fuerte es el razonamiento sobre código con contexto amplio: puedes pegarle ficheros completos, tracebacks largos y fragmentos de la documentación de PyQGIS en el mismo mensaje y mantiene coherencia. Especialmente útil para diseño de arquitectura, refactoring y debugging donde el error no es obvio. Es el modelo que mejor se comporta cuando le das un plugin entero y le pides que entienda qué está pasando.

**GPT-4o**
Muy competente en Python general y tiene buena "memoria" de patrones PyQGIS comunes por volumen de entrenamiento. Las completaciones en Copilot/Cursor con GPT-4o son rápidas y acertadas para código repetitivo (conectar señales Qt, poblar QComboBox, gestionar rutas). Menos sólido que Claude en razonamiento multi-fichero largo.

**Gemini 2.5 Pro**
Ventana de contexto muy grande (hasta 1M tokens). Útil puntualmente si necesitas analizar toda la documentación INSPIRE + varios ficheros a la vez. En generación de código PyQGIS es menos consistente que los dos anteriores, pero ha mejorado notablemente.

**Modelos locales (Qwen2.5-Coder, DeepSeek-Coder v2)**
Si tienes GPU decente, aportan privacidad total y cero coste marginal. Para Python general son sorprendentemente buenos. Para PyQGIS específico flojean: la API de QGIS es un nicho pequeño en sus datos de entrenamiento y alucinan más métodos. Útiles para tareas genéricas (parsear XML, manipular ZIPs, lógica de negocio pura).

## El consejo más práctico

El mayor riesgo con cualquier modelo en PyQGIS no es el modelo en sí: es que **todos tienden a generar código plausible con métodos deprecados o inexistentes** en versiones recientes de QGIS. GPT-4o lo hace, Claude lo hace, todos lo hacen.

La defensa es siempre la misma: tener abierta la [documentación PyQGIS](https://qgis.org/pyqgis/master/) y la [PyQGIS Developer Cookbook](https://docs.qgis.org/latest/en/docs/pyqgis_developer_cookbook/) como referencia, y contrastar cualquier método que no reconozcas antes de ejecutarlo. Ningún modelo sustituye esa verificación.

Dicho esto, para empezar con un solo modelo: **Claude dentro de Cursor** cubre el 80% de los casos de este plugin sin necesidad de alternar.
